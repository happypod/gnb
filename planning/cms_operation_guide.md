# GNB Planning 웹사이트 - CMS 구조 & 운영 가이드 (v1.0)

본 문서는 웹사이트가 단순한 홍보 채널을 넘어, **회사의 실적과 지적 자산이 누적되는 '디지털 아카이브'**로 기능하도록 CMS(콘텐츠 관리 시스템)의 논리적 구조와 운영 규칙을 정의합니다.

---

## 📄 A. CMS 콘텐츠 유형 정의표 (Content Types)

| 콘텐츠 ID | 유형명 (Name)      | 정의 및 용도                      | 주요 특징                                                |
| :-------- | :----------------- | :-------------------------------- | :------------------------------------------------------- |
| **CT-01** | **Project** (핵심) | 수행한 용역/컨설팅 실적 데이터    | 지도 좌표, 성과 수치 포함. 타 콘텐츠와 가장 많이 연결됨. |
| **CT-02** | **Expertise**      | 사업 분야 (어촌/농촌 등) 카테고리 | 프로젝트를 분류하는 기준이자, GNB의 업역을 정의.         |
| **CT-03** | **Insight**        | 연구 보고서, 칼럼, 정책 해설      | 전문성을 증명하는 지적 자산. 검색 유입의 핵심.           |
| **CT-04** | **News**           | 보도자료, 현장 스케치 가벼운 소식 | 시의성 있는 콘텐츠. 외부 링크 연결 위주.                 |
| **CT-05** | **People**         | 연구원 및 전문가 프로필           | "누가 수행했는가?"를 증명. 프로젝트와 1:N 연결.          |
| **CT-06** | **Page**           | 회사소개, 채용 등 정적 페이지     | 빈번한 수정 없음. 브랜드 메시지 전달.                    |

## 📄 B. 콘텐츠 유형별 필드 구조표 (Schema)

### 1. Project (프로젝트 - 핵심 자산)

_목적: 지도 연동 및 신뢰 검증용 데이터 정동_

| 필드명           | 데이터 타입   | 필수 | 설명 및 용도                                          |
| :--------------- | :------------ | :--- | :---------------------------------------------------- |
| **Title**        | Text          | O    | 사업명 (예: 부안군 곰소항 어촌뉴딜 300 기본계획)      |
| **Location**     | Region Select | O    | 광역/기초 지자체 구분 (필터링 및 지도 핀 위치 데이터) |
| **Coordinates**  | Geo Point     | O    | 위도/경도 (지도 위에 핀을 찍기 위함)                  |
| **Department**   | Relation      | O    | **[Expertise]**와 연결 (농촌/어촌/지역개발 등)        |
| **Duration**     | Date Range    | O    | 시작일 ~ 종료일 (진행중/완료 상태 자동 구분)          |
| **Key Result**   | Text (Short)  | O    | **검증용 핵심 성과 한 줄** (예: 방문객 200% 증가)     |
| **Cover Image**  | Image         | O    | 카드 썸네일 및 헤더 배경 (현장 사진 권장)             |
| **Participants** | Relation      | △    | **[People]**과 연결 (수행 책임자/참여자)              |
| **Description**  | Rich Text     | O    | 문제 -> 해결 -> 성과의 구조화된 상세 본문             |
| **Is Public**    | Boolean       | O    | 공개 여부 (보안 유지 필요한 사업은 비공개 아카이빙)   |

## 📄 D. 콘텐츠 등록·승인 흐름도 (Workflow)

### 시나리오 A: 신규 프로젝트("어촌신활력 00사업") 등록 시

1.  **[Contributor/Editor] 초안 작성**
    - 현장에서 찍은 사진 업로드.
    - 사업명, 기간, 개요 등 기초 Fact 입력.
    - 상태: `Draft` (임시저장)
2.  **[Editor] 성과/검증 데이터 보완**
    - 'Key Result' (정량 성과) 입력.
    - 문장 톤앤매너 수정 (감성적 표현 제거).
    - 관련 전문가(People) 태깅.
    - 액션: `Request Review` (검토 요청)
3.  **[Admin] 최종 승인**
    - 대외비 정보 포함 여부, 오탈자 확인.
    - 액션: `Publish` (공개)
    - 결과: **지도에 핀 생성됨 + 포트폴리오 리스트 최상단 노출 + 메인 카운터 숫자 자동 증가.**

### 시나리오 B: 정책 칼럼 등록 시

1.  **[Contributor]** 자료 조사 및 초안 작성 -> `Draft`
2.  **[Editor]** 전문성 검증 및 교정 -> `Publish` (칼럼은 Editor 전결 가능하도록 설정)

## E. 유지관리 & 확장 전략

1.  **자동화된 자산화:**
    - 프로젝트를 등록하면 `지역별`, `유형별` 통계가 자동으로 갱신되도록 설계합니다. (예: "전남 지역 수행 1위" 같은 데이터를 언제든 추출 가능)
2.  **관계형 연결 (Relational Link):**
    - [프로젝트] 상세 페이지 하단에 "이 사업을 수행한 [전문가]" 카드가 자동 노출됩니다.
    - 반대로 [전문가] 상세 페이지에는 "참여한 [프로젝트]" 리스트가 자동 노출됩니다.
    - 이는 수동 관리를 줄이고 사이트의 밀도를 높여줍니다.
3.  **데이터 시각화 확장성:**
    - 초기에는 단순 핀으로 지도에 표시하지만, 향후 데이터가 500건 이상 쌓이면 `Heatmap`(밀도 지도) 등으로 시각화 방식을 변경할 수 있도록 `Coordinates`(좌표) 필드를 필수로 관리합니다.
