# GNB Planning 웹사이트 - 디자인 시스템 & 톤 가이드 (v1.0)

본 문서는 GNB Planning의 브랜드 정체성인 **'신뢰(Trust)'와 '현장(Field)'**을 시각 언어와 텍스트로 일관되게 전달하기 위한 **결정 기준 문서(Rulebook)**입니다. 디자이너나 개발자가 바뀌어도 흔들리지 않는 원칙을 정의합니다.

---

## 📄 A. 디자인 시스템 원칙 요약 (Core Principles)

### 1. 핵심 키워드 (5 Pillars)

- **Trust (신뢰):** 과장하지 않고, 보여줄 수 있는 것만 보여준다.
- **Restraint (절제):** 장식 요소를 최소화하여 콘텐츠(실적)가 주인공이 되게 한다.
- **Field-oriented (현장성):** 책상 위 보고서가 아닌, 흙 묻은 현장의 생동감을 우선한다.
- **Structured (구조화):** 모든 정보는 위계가 명확해야 하며, 훑어만 봐도 요점을 알 수 있어야 한다.
- **Scalable (확장성):** 프로젝트가 100개에서 1000개로 늘어나도 디자인이 깨지지 않아야 한다.

### 2. 접근성 & 공공 UX 원칙

- **High Contrast (고대비):** 야외 현장에서 모바일로 접속하는 주민과 노안이 온 고령층 공무원을 고려하여, 배경과 텍스트 명도 대비를 확실하게 둔다.
- **No Color Dependency:** 색맹/색약 사용자를 위해 '색상'만으로 정보를 구분하지 않고, '텍스트 라벨'이나 '아이콘'을 병행한다.
- **Touch Friendly:** 모바일에서 손가락이 굵은 어르신도 쉽게 누를 수 있도록 터치 영역(Target Size)을 충분히 확보한다 (최소 44px).

## 📄 B. 컬러·타이포·레이아웃 역할 정의표

### 1. Color System Roles (값 지정 금지)

| 구분          | 역할 (Role) | 사용 목적                              | 사용 금지 / 주의사항                                                    |
| :------------ | :---------- | :------------------------------------- | :---------------------------------------------------------------------- |
| **Primary**   | Identity    | 로고, GNB 네비게이션, 핵심 브랜드 요소 | 형광색 등 눈이 아픈 고채도 금지. 텍스트 가독성을 해치는 배경 사용 금지. |
| **Secondary** | Information | 섹션 배경, 비활성 버튼, 보조 정보 구분 | Primary와 혼동될 만큼 강한 색 사용 금지.                                |
| **Accent**    | Attention   | CTA 버튼, 핵심 데이터(수치), 상태 알림 | 페이지당 1~2회로 제한. 남발 시 '경고' 처럼 보여 사용자가 피로감을 느낌. |
| **Neutral**   | Base        | 본문 텍스트, 배경색, 라인, 카드 그림자 | 완전한 #000(Black) 대신 아주 짙은 회색 권장 (눈의 피로 감소).           |

### 2. Typography Hierarchy

| 계층        | 역할 (Role) | 사용 위치                                    | 규칙 (Rules)                                                           |
| :---------- | :---------- | :------------------------------------------- | :--------------------------------------------------------------------- |
| **Display** | Impact      | 메인페이지 Hero 섹션, 아주 짧은 캐치프레이즈 | 장식체(손글씨 등) 절대 금지. 자간을 좁혀 단단한 인상을 줄 것.          |
| **Heading** | Structure   | 페이지 제목, 섹션 타이틀, 카드 제목          | 본문과 명확히 구분되는 굵기(Weight) 사용. 명조 계열 지양 (고딕 권장).  |
| **Body**    | Value       | 상세 설명, 프로젝트 개요, 일반 텍스트        | 가독성이 최우선. 줄글은 줄당 40~60자 내외로 끊어 호흡 조절.            |
| **Caption** | Detail      | 사진 캡션, 출처, 태그, 날짜                  | 작아도 뭉개지지 않는 폰트 사용. 연한 회색 사용 시 명도 대비 체크 필수. |

### 3. Layout & Grid Principles

- **Bento Grid:** 정보를 직사각형 블록(Card) 단위로 모듈화하여 배치. 복잡한 정보도 정동된 느낌을 줌.
- **One Message Per View:** 스크롤을 멈췄을 때, 한 화면에는 하나의 핵심 메시지만 보이도록 여백을 충분히 둠.
- **Mobile First Stack:** 데스크탑의 가로 배치가 모바일에서는 자연스럽게 세로 스택(Stack)으로 쌓이는 유동적 구조.

## 📄 C. UI 컴포넌트 사용 가이드 표

| 컴포넌트   | 정의 & 사용 원칙                                          | 금지 패턴 (Don't)                                      | 공공 신뢰 관점 주의사항                                           |
| :--------- | :-------------------------------------------------------- | :----------------------------------------------------- | :---------------------------------------------------------------- |
| **Card**   | 정보를 담는 기본 컨테이너. 그림자나 테두리로 구분감 제공. | 내부에 또 다른 박스를 중첩하여 복잡도 증가 금지.       | 너무 둥근 모서리(Playful)보다는 적당한 라운드(Professional) 권장. |
| **Button** | 사용자의 명확한 행동(Action)을 유도.                      | 텍스트 링크인지 버튼인지 모호하게 디자인 금지.         | "알아보기" 같은 모호한 레이블 대신 "사업 문의하기" 처럼 명확하게. |
| **Tag**    | 키워드 분류 #태그.                                        | 한 줄을 넘어가서 줄바꿈 되도록 길게 나열 금지.         | 알록달록한 무지개색 사용 금지 (산만함).                           |
| **Filter** | 데이터 탐색 도구.                                         | 선택된 항목과 선택되지 않은 항목의 구분이 모호한 경우. | 내가 지금 무엇을 보고 있는지(Current State) 항상 표시해야 함.     |
| **Badge**  | 상태 표시 (진행중, 완료 등).                              | 텍스트와 배경색 대비가 낮아 안 읽히는 경우.            | 직관적인 신호등 컬러(초록=완료, 회색=대기) 활용 권장.             |
| **Image**  | 증거 자료.                                                | 저해상도, 워터마크가 찍힌 이미지 사용.                 | '연출된 웃는 모델' 사진 절대 금지. **땀 흘리는 현장 사진** 우선.  |

## 📄 D. 카피 톤 & 금지어 리스트

### 1. 기본 톤앤매너 (Tone & Manner)

- **단정함 (Polite):** "~요" 대신 "~니다"를 쓰되, 딱딱한 공문서보다는 부드럽게.
- **설명형 (Descriptive):** 감정을 호소하기보다 사실 관계를 명확히 서술.
- **자신감 (Confident):** 겸손하되 비굴하지 않게. "노력하겠습니다"보다 "**해결했습니다**" 사용.

### 2. 주어 사용 원칙

- **Bad:** "우리는 최고의 전문가 그룹으로서..." (주어=우리)
- **Good:** "현장의 목소리는 데이터로 증명됩니다." (주어=현장/사물) -> 객관성 확보.

### 3. 단어 가이드 (White/Black List)

| 구분       | 권장 표현 (Do)                         | 금지 표현 (Don't)                    | 이유                                       |
| :--------- | :------------------------------------- | :----------------------------------- | :----------------------------------------- |
| **서술어** | 수행했다, 변화했다, 해결했다, 달성했다 | 보여진다, 생각된다 (피동형)          | 책임감 있고 주도적인 태도를 보여주기 위함. |
| **수식어** | 구체적인, 실증된, 현장의, 주민 주도    | 최고의, 혁신적인, 선도하는, 완벽한   | 근거 없는 최상급 표현은 신뢰도를 떨어뜨림. |
| **연결어** | 그리하여, 따라서 (논리적 접속사)       | 그런데, 하지만, 솔직히 (감정적/전환) | 논리적인 흐름을 유지해야 함.               |

2.  **지속성:** 유행 타는 그라데이션이나 3D 그래픽을 배제하고 '구조' 중심이라 오래가는가? **(YES)**
3.  **수용성:** 50대 공무원이 봐도 "가볍지 않고 진중하다"고 느낄 수 있는가? **(YES)**
4.  **확장성:** 나중에 '연구 실적' 페이지가 추가되어도 Bento Grid와 카드 시스템을 재활용 가능한가? **(YES)**
