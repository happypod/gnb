# GNB Planning 웹사이트 - 대표 사업 상세 페이지 UX 설계 (v1.0)

본 문서는 GNB Planning의 프로젝트 수행 능력을 검증하는 '상세 페이지(Case Study)'의 표준 UX 구조를 정의합니다. 단순한 실적 보고서가 아닌, "문제 해결 능력"을 증명하는 콘텐츠 포맷입니다.

---

## 📄 A. 상세 페이지 UX 구조도

### 핵심 컨셉: "문제(Problem) → 개입(Intervention) → 변화(Change) → 검증(Verification)"

| 순서 | 섹션명             | 목적 (Purpose)                                     | 주요 타겟              |
| :--- | :----------------- | :------------------------------------------------- | :--------------------- |
| 1    | **Project Header** | 5초 내 사업 개요(규모, 성격) 파악                  | 공무원 (상급자 보고용) |
| 2    | **Background**     | 사업의 필요성과 당시의 **난제(Challenge)** 공유    | 공무원 (공감)          |
| 3    | **GNB's Role**     | 단순 용역 대행이 아닌 **주도적 해결사** 역할 강조  | 추진위, 공무원         |
| 4    | **Process**        | 현장에서의 **구체적 실행 과정** 시각화             | 주민, 연구원           |
| 5    | **Outcome**        | **정량적/정성적 데이터**로 성공 증명               | 공무원 (감사 대비)     |
| 6    | **Field Evidence** | 책상이 아닌 **현장**에서 일했음을 인증 (사진/문서) | 신뢰성 검증            |
| 7    | **Applicability**  | "우리 지역에도 적용 가능할까?"에 대한 확신 부여    | 잠재 발주처            |
| 8    | **Next Action**    | 유사 사례 탐색 또는 컨설팅 문의 유도               | 전환 (Conversion)      |

---

## 📄 B. 섹션별 필수/선택 정보 목록

### 1. Project Header (필수)

- **Project Name:** [지역명] 사업명 (예: 부안군 곰소항 어촌뉴딜 300 기본계획)
- **Location:** 행정구역 (예: 전북 부안군 진서면)
- **Category:** 사업 유형 (어촌 / 농촌 / 지역개발 / 연구 등)
- **Duration:** 수행 기간 (YYYY.MM ~ YYYY.MM) -> **완료 여부 중요**
- _(선택)_ **Client:** 발주처 (로고 또는 텍스트)

### 2. Background (필수)

- **Challenge (핵심 문제):**
  - _행정 관점:_ "낙후된 시설로 인한 안전 문제 및 관광 경쟁력 저하"
  - _주민 관점:_ "소득 감소로 인한 청년 이탈 및 마을 활력 저하"
- **Goal:** 사업을 통해 달성하고자 했던 최종 목표 1문장.

### 3. GNB's Role (필수)

- **Core Tasks:** GNB가 수행한 핵심 역할 리스트 (아이콘 + 단어)
  - 예: [기획] 마스터플랜 수립, [조정] 주민 협의체 운영, [실행] 앵커시설 시범 운영
- **Key Approach:** 문제를 해결하기 위해 도입한 GNB만의 접근 방식.
  - 예: "일방적 설계가 아닌, 10회 이상의 주민 워크숍을 통한 상향식 계획 수립"

### 4. Process (선택 - 복잡한 사업일 경우 필수)

- **Timeline:** 주요 마일스톤 (착수 -> 중간 -> 최종 성과보고)
- **Activity:** 각 단계별 주요 사진 또는 산출물 썸네일.

### 5. Outcome (필수 - 가장 중요)

- **Before / After:** 가능하다면 시설물 또는 마을 전경 비교 사진.
- **Quantitative (정량):**
  - 방문객 수 변화 (예: 연 1만명 -> 3만명)
  - 주민 소득 증대 (예: 특산물 판매액 150% 증가)
  - 시설 가동률 등
- **Qualitative (정성):**
  - 주민 인터뷰/반응 "예전엔 어두었는데 밤에도 사람이 다녀요."
  - 행정 평가 "우수 사례 선정" 등

### 6. Field Evidence (필수)

- **Gallery:** 현장 활동 사진 (주민과 회의하는 모습, 현장 답사 모습 등 3~4장)
- _(선택) 문서:_ 착수보고서 표지, 협약서 등 '공식 문서' 이미지.

### 7. Applicability (선택)

- **Suggestion:** "이런 지역에 추천합니다."
  - 예: "인구 소멸 위기 4단계 지역", "특산물은 있으나 판로가 없는 어촌"

## 📄 C. 공무원·주민 시점 UX 체크리스트

### ☑️ 공무원(발주처) 시나리오 점검

1.  **[보고 용이성]** 상급자에게 화면을 띄워놓고 "이 회사가 여길 이렇게 바꿨답니다"라고 1분 브리핑이 가능한가?
    - -> **YES.** Header(개요)와 Outcome(결과)만 읽어도 보고 가능.
2.  **[역량 검증]** "그냥 말만 잘하는 거 아니야?"라는 의심을 Process와 Evidence 섹션이 해소하는가?
    - -> **YES.** 현장 사진과 구체적인 역할 서술로 '현장성' 입증.
3.  **[유사성 확인]** Applicability 섹션을 통해 우리 군의 상황과 매칭할 수 있는가?
    - -> **YES.** 적용 가능 조건을 명시하여 도입 타당성 확보.

### ☑️ 주민/추진위원장 시나리오 점검

1.  **[이해 용이성]** 어려운 전문 용어(SW, HW, 앵커조직 등) 도배 대신 '변화된 모습' 위주인가?
    - -> **YES.** Background(주민 관점 문제)와 Outcome(주민 반응)에 쉬운 언어 배치.
2.  **[신뢰 형성]** "우리 의견을 잘 들어줄까?"에 대한 답이 있는가?
    - -> **YES.** GNB's Role과 Field Evidence에서 '소통'과 '주민 참여' 강조.

## ✅ 완료 기준 (Self Check)

- 보고서 없이 사업 이해 가능? **(O)** -> "문제-해결-결과" 스토리라인 구축.
- GNB 역할 명확? **(O)** -> 단순 나열이 아닌 '접근 방식'과 '개입' 명시.
- 감성보다 신뢰? **(O)** -> 홍보 문구 배제, 데이터와 현장 증거 중심.
- 재사용 가능? **(O)** -> 모든 유형(어촌/농촌 등)에 대입 가능한 표준 모듈형 구조.
