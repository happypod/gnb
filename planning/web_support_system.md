# GNB Planning 웹사이트 - 웹 기반 제안·보고 보조 시스템 설계 (v1.0)

본 문서는 GNB Planning 웹사이트가 단순한 홍보 채널을 넘어, 내부 임직원의 **제안서 작성(Proposal)과 성과 보고(Report) 업무를 획기적으로 줄여주는 '업무 보조 시스템'**으로 기능하도록 설계한 전략 문서입니다.

---

## 📄 A. 웹 기반 제안·보고 보조 시스템 개념도

### 핵심 철학: "Web is the Archive, Doc is the Summary."

- **웹(Web):** 모든 프로젝트의 상세 과정, 현장 사진, 구체적 성과 데이터가 저장된 **'방대한 근거 저장소(Repository)'**
- **문서(Doc):** 웹에 있는 방대한 자료 중, 해당 발주처가 듣고 싶어 하는 핵심만 뽑아낸 **'요약본(Summary)'**

### 4대 핵심 기능 (Concept)

1.  **근거 페이지 (Proof Node):** 제안서에 한 줄로 적힌 "유사 실적 보유"라는 주장을 뒷받침하는 상세 증거 페이지. (Project Detail)
2.  **설명 블록 (Explanation Block):** 복잡한 성과를 "방문객 200% 증가"처럼 직관적인 숫자와 그래프로 시각화해 둔 모듈. 제안서에 그대로 캡처해 쓸 수 있음.
3.  **제안 세트 (Propose Package):** 특정 사업 유형(예: 어촌뉴딜) 제안 시 묶어서 링크를 보내면 좋은 페이지 조합.
    - _Set:_ [지도(전국)] + [유사 사례(전남)] + [전문가(OOO 박사)]
4.  **보고 흐름 (Report Flow):** "문제 -> 개입 -> 성과"로 이어지는 웹의 스크롤 순서가 곧 발표 자료의 목차(Index)가 되도록 설계.

## 📄 B. 제안 단계별 웹 활용 시나리오 표

| 단계                  | 기존 업무 방식 (Before)                                   | 웹 활용 방식 (After)                                                              | 기대 효과                                                     |
| :-------------------- | :-------------------------------------------------------- | :-------------------------------------------------------------------------------- | :------------------------------------------------------------ |
| **1. 제안서 작성**    | 과거 제안서 폴더를 뒤져서 유사 실적 텍스트 복사/붙여넣기. | **[Portfolio]**에서 조건 검색 후, 해당 **[상세 페이지]**의 핵심 성과/사진을 발췌. | 'Ctrl+C/V' 반복 업무 감소. 최신 데이터(Update) 반영 보장.     |
| **2. 발표(PT) 준비**  | PPT 장표마다 실적 사진 10장씩 때려 넣음. (용량 폭증)      | PPT에는 핵심 사진 1장만 넣고, **"상세 내용은 QR코드/링크 참조"**로 대체.          | PPT 장수 대폭 감소. 심사위원이 원할 때만 깊게 확인 가능.      |
| **3. 질의응답 (Q&A)** | 예상 질문에 대비한 별첨 자료(Appendix) 수십 장 출력.      | 질의 시 **[Insight]**나 **[Data]** 페이지를 화면에 띄워 실시간 설명.              | "자료가 없어서 확인해 보겠습니다"라는 답변 방지. 전문성 어필. |
| **4. 완료 보고**      | 결과보고서 책자 만드느라 며칠 밤샘.                       | 웹사이트 **[Project Detail]** 업데이트 후, 그 링크를 보고서 요약본과 함께 제출.   | '성과 확산' 실적으로 인정받음. (디지털 아카이브 구축 성과)    |

## 📄 C. 내부 실무자 사용 가이드 (운영 원칙)

### Q1. 제안서에 웹 링크는 언제 쓰는가?

- **A.** "지면이 부족할 때" 그리고 "자랑할 증거가 너무 많을 때" 씁니다.
  - _Good:_ "이 사업 외에도 15건의 유사 실적이 있습니다. (전체 리스트 보기 > Link)"
  - _Bad:_ "우리 회사 소개입니다. (홈페이지 > Link)" -> 성의 없어 보임.

### Q2. 웹과 문서 중 무엇이 '정본(Canonical)'인가?

- **A.** **웹(Web)이 최신 정본입니다.**
  - 문서는 작성 시점(2024년 5월)의 스냅샷일 뿐입니다. 웹사이트는 2024년 12월의 성과까지 계속 업데이트됩니다. 따라서 실적 수치는 항상 웹을 기준으로 작성하세요.

### Q3. 링크는 몇 개까지 허용하는가?

- **A.** 제안서 하나당 **최대 3개(Key 3)**로 제한합니다.
  - 너무 많은 QR코드는 평가위원을 산만하게 만듭니다. 가장 강력한 '한 방'(대표 실적 상세 페이지) 위주로 배치하세요.

## 📄 D. 웹–문서 역할 분리 원칙 정리표

| 구분         | 웹 (Web)                          | 제안서/보고서 (Doc)       | 분리 원칙 (Rule)                                                                     |
| :----------- | :-------------------------------- | :------------------------ | :----------------------------------------------------------------------------------- |
| **목적**     | **축적(Archive)** & 검색          | **설득(Persuade)** & 요약 | 웹에 있는 걸 문서에 다 쓰지 마라. 문서에 쓴 걸 웹에 다 올리지 마라.                  |
| **호흡**     | 긴 호흡 (TMI 허용)                | 짧은 호흡 (두괄식)        | 문서는 결론부터, 웹은 과정부터 보여준다.                                             |
| **업데이트** | 수시 (새로운 사진/성과 즉시 반영) | 고정 (제출일 기준)        | 제안서 제출 후 웹을 몰래 수정하여 혼란을 주지 않는다. (제안 시점 버전 PDF 백업 필수) |
| **대상**     | 불특정 다수 (주민 포함)           | 특정 평가위원 (3~5명)     | 보안 사항(예산, 개인정보)은 문서에만 담고, 웹에는 절대 올리지 않는다.                |

## ⚠️ 리스크 및 오해 방지 설계

1.  **불일치 대응:** "제안서엔 5건인데 웹엔 4건이네요?"
    - -> 방어 논리: "웹은 '공개 동의'된 건만 엄선했습니다. 보안 유지를 위해 제안서에만 전체 실적을 수록했습니다." (사전 교육 필수)
2.  **과다 노출:** "경쟁사가 우리 웹 보고 전략 베끼면 어쩌죠?"
    - -> 대응: '방법론(Methodology)' 페이지는 너무 구체적인 노하우 대신 '철학'과 '프로세스' 위주로 서술하여 핵심 영업비밀 보호.

## ✅ 완료 기준 (Self Check)

1.  **업무 감소:** 제안서 쓸 때 "어디 있더라?" 찾는 시간이 줄었는가? **(YES)** -> 웹에서 검색하면 나옴.
2.  **실용성:** 직원들이 "웹사이트 좀 업데이트해주세요, 제안서 써야 해요"라고 먼저 말하게 되는가? **(YES)** -> 웹이 곧 내 무기가 되니까.
3.  **안전성:** 공공 발주처가 싫어할 만한 요소(과한 홍보, 보안 누설)가 차단되었는가? **(YES)** -> 역할 분리 원칙 준수.
